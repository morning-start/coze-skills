# 场景：修改技能

## 适用场景

修改已发布的技能，包括修复错误、添加功能、调整结构。

---

## 快速决策

```
变更类型判断：
│
├─ 仅修复错误（错别字、描述澄清）
│   └─ Fix → patch +1
│
├─ 添加新功能（向后兼容）
│   └─ Feature → minor +1
│
└─ 破坏性变更（接口修改、能力删除）
    └─ Breaking → major +1
```

---

## 第一步：查阅信息

### 操作

**1. 检查当前状态**

```bash
head -10 SKILL.md
```

**2. 确定变更类型**

| 变更类型 | 说明 | 版本递增 | 示例 |
|---------|------|---------|------|
| **Fix** | 修复错误 | patch +1 | 错别字、澄清描述 |
| **Feature** | 新增功能 | minor +1 | 添加能力、新示例 |
| **Breaking** | 破坏性变更 | major +1 | 修改接口、删除能力 |

**3. 版本决策**

```
是否有破坏性变更？
├── 是 → major +1 (v1.0.0 → v2.0.0)
└── 否 → 是否有新功能？
    ├── 是 → minor +1 (v1.0.0 → v1.1.0)
    └── 否 → patch +1 (v1.0.0 → v1.0.1)
```

---

## 第二步：执行操作

### 开发阶段

**准入**: 变更需求明确，版本策略已定

**操作**:

1. 备份原文件
   ```bash
   cp SKILL.md SKILL.md.bak
   ```

2. 执行修改
   - 修改目标内容
   - 同步更新关联内容

3. 更新版本号

   ```yaml
   # 原版本 + 变更类型 = 新版本
   version: v1.1.0
   ```

**准出**:
- [ ] 修改内容完成
- [ ] 版本号已更新

### 测试阶段

**准入**: 开发阶段完成

**操作**:

1. 变更影响检查
   - [ ] 修改后技能仍可用
   - [ ] 未引入新问题

2. 标准化检验
   ```bash
   head -10 SKILL.md
   grep "version:" SKILL.md
   ```

**准出**:
- [ ] 变更影响已评估
- [ ] 标准化检验通过

### 发布阶段

**准入**: 测试阶段完成

**操作**:

```bash
git add SKILL.md
git commit -m "feat(<skill>): <变更说明>"
git tag -a v1.1.0 -m "Release v1.1.0"
```

## 第三步：检查验收

### 版本更新检查

- [ ] 版本号已更新
- [ ] 版本号格式正确
- [ ] 版本递增合理

### 兼容性检查

- [ ] 向后兼容（如需要）
- [ ] 接口变更已说明（如有）
- [ ] 破坏性变更已标记（如有）

---

## 版本更新详解

### Patch (v1.0.0 → v1.0.1)

适用：修复错别字、澄清描述、修正示例错误

### Minor (v1.0.0 → v1.1.0)

适用：添加新能力、添加新示例、扩展使用场景

### Major (v1.0.0 → v2.0.0)

适用：修改接口定义、删除核心能力、重大架构调整

---

## 常见问题

**Q: 如何确定变更类型？**
- 仅修复错误 → Fix
- 添加新功能（向后兼容） → Feature
- 破坏性变更 → Breaking

**Q: 发现已发布版本有错误？**
- 创建新的 patch 版本修复
