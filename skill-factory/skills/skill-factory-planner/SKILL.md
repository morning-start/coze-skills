# Skill Factory Planner - 技能规划器

## 技能职责边界

**本技能负责**：
- 分析报告中的模块划分
- 设计母技能和子技能的结构
- 分析技能间的依赖关系
- 生成可执行的技能生成计划

**本技能不负责**：
- 读取原始技术内容（由 analyzer 负责）
- 生成具体技能文件（由 generator 负责）
- 验证打包结果（由 packager 负责）

---

## 输入判断

### 有效输入

analyzer 输出的技术分析报告，必须包含：
- 技术名称
- 核心功能（至少 1 个）
- 关键概念（至少 2 个）

### 无效输入处理

- 报告缺少必填字段 → 返回错误："分析报告不完整，缺少 {字段名}，请返回 analyzer 重新分析"
- 报告完整度 < 60% → 返回错误："分析报告完整度过低（{完整度}%），无法进行有效规划，请先补充分析"
- 输入不是分析报告格式 → 返回错误："输入不是有效的分析报告格式"

---

## 核心逻辑

### 工作流程

```
分析报告 → 模块识别 → 技能拆分 → 依赖分析 → 执行计划 → 输出
                        ↓
                  拆分是否合理？
                   ↓         ↓
                  是        否
               继续      调整拆分
```

### 步骤详解

#### 步骤 1：模块识别

基于分析报告中的"核心功能"和"关键概念"识别独立模块：

**识别规则**：
- 每个核心功能 → 至少 1 个子技能
- 每 3-4 个相关关键概念 → 可合并为 1 个子技能
- 所有子技能的共性内容 → 归入母技能

**模块类型判断**：

| 核心功能数量 | 建议子技能数量 | 示例 |
|--------------|----------------|------|
| 1-2 个 | 1 个子技能 | 单一工具 |
| 3-5 个 | 2-3 个子技能 | 中型框架 |
| 6+ 个 | 4+ 个子技能 | 大型平台 |

#### 步骤 2：技能拆分设计

**母技能设计**：
1. 命名：统一为 `{技术名}-family` 或 `{技术名}-skill`
2. 职责：协调子技能、提供概览、定义依赖关系
3. 内容：技能定位、工作流、触发条件

**子技能设计**：
1. 命名：使用 kebab-case，如 `vue-core`、`vue-router`
2. 职责：每个子技能职责单一
3. 优先级：根据依赖关系确定生成顺序

**拆分合理性判断**：

| 判断项 | 合理标准 | 不合理时的调整 |
|--------|----------|---------------|
| 子技能数量 | 1-8 个 | 过多则合并，过少则拆分 |
| 子技能粒度 | 一个子技能能在 200 行内描述清楚 | 过大则拆分，过小则合并 |
| 技能命名 | 名称能反映核心功能 | 重命名 |
| 职责划分 | 无重叠内容 | 重新划分边界 |

#### 步骤 3：依赖关系分析

**依赖分析规则**：
1. 找出所有无依赖的技能（基础技能）→ 可最先生成
2. 找出有依赖的技能 → 必须在依赖技能之后生成
3. 检测循环依赖 → 有则报错，返回拆分设计

**依赖关系表示**：

```markdown
## 技能依赖图

skill-1 (无依赖)
skill-2 (依赖 skill-1)
skill-3 (依赖 skill-1, skill-2)
```

#### 步骤 4：执行计划生成

根据依赖关系生成串行和并行计划：

**串行组**（必须按顺序执行）：
```
组 1：无依赖技能 → 组 2：依赖组1的技能 → 组 3：依赖组1/2的技能 → ...
```

**并行组**（同一组内可并行）：
```
组 2：skill-2, skill-3 (都只依赖 skill-1)
```

## 输出标准

### 合格计划

必须包含：
- 母技能命名和定位
- 每个子技能的命名、定位、职责
- 完整的依赖关系图
- 可执行的生成顺序

### 输出格式

```markdown
## 技能拆分计划

### 技术名称
{技术名称}

### 母技能设计
- **命名**：{母技能名}
- **定位**：{技能族编排器/单一技能}
- **核心职责**：{3-5 点}

### 子技能设计

#### 子技能 1：{子技能名}
- **对应模块**：{来自分析报告的哪些功能/概念}
- **技能定位**：{基础/核心/高级}
- **核心职责**：{1-3 点}
- **依赖技能**：无 / {技能名}

#### 子技能 2：{子技能名}
...

### 依赖关系图

```
{技能名} (无依赖)
  ↓
{技能名} (依赖 {技能名})
  ↓
{技能名} (依赖 {技能名}, {技能名})
```

### 执行计划

| 顺序 | 技能 | 可并行组 |
|------|------|----------|
| 1 | {子技能1} | - |
| 2 | {子技能2}, {子技能3} | 组 1 |
| 3 | {子技能4} | - |

### 复杂度评估
- **子技能数量**：X 个
- **最大依赖深度**：X 层
- **预计生成时间**：{短/中/长}
```
---

## 失败处理

| 失败场景 | 处理方式 |
|----------|----------|
| 分析报告不完整 | 返回错误，要求补充分析 |
| 子技能数量超出范围 (1-8) | 提示合并或拆分建议，等待确认 |
| 存在循环依赖 | 返回错误，列出循环路径，要求重新设计拆分 |
| 依赖关系不明确 | 保守处理：将有疑问的技能标记为依赖其他技能，优先保证无环 |
| 分析报告完整度 < 60% | 返回错误，建议返回 analyzer 补充 |

## 使用示例

**输入**：Vue.js 技术分析报告

**输出**：
```markdown
## 技能拆分计划

### 技术名称
Vue.js

### 母技能设计
- **命名**：vue-family
- **定位**：技能族编排器
- **核心职责**：
  - 协调子技能完成 Vue.js 技能族构建
  - 提供 Vue.js 整体学习路径
  - 定义子技能依赖关系

### 子技能设计

#### 子技能 1：vue-core
- **对应模块**：响应式、组件系统、指令
- **技能定位**：基础
- **核心职责**：
  - Vue 实例创建和生命周期
  - 响应式数据绑定原理
  - 指令系统使用
- **依赖技能**：无

#### 子技能 2：vue-composition
- **对应模块**：Composition API
- **技能定位**：核心
- **核心职责**：
  - setup 函数用法
  - 组合式函数封装
  - 响应式引用
- **依赖技能**：vue-core

#### 子技能 3：vue-router
- **对应模块**：路由管理
- **技能定位**：核心
- **核心职责**：
  - 路由配置
  - 导航守卫
  - 嵌套路由
- **依赖技能**：vue-core

#### 子技能 4：vue-state
- **对应模块**：状态管理
- **技能定位**：高级
- **核心职责**：
  - Pinia/Vuex 使用
  - 状态持久化
  - 模块化设计
- **依赖技能**：vue-core

### 依赖关系图

```
vue-core (无依赖)
  ↓
vue-composition (依赖 vue-core)
vue-router (依赖 vue-core)
  ↓
vue-state (依赖 vue-core)
```

### 执行计划

| 顺序 | 技能 | 可并行组 |
|------|------|----------|
| 1 | vue-core | - |
| 2 | vue-composition, vue-router | 组 1 |
| 3 | vue-state | - |

### 复杂度评估
- **子技能数量**：4 个
- **最大依赖深度**：2 层
- **预计生成时间**：中
```
