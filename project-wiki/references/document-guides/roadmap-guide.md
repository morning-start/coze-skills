## 概览

### 定义

**ROADMAP.md** 是项目发展规划文档，记录未来版本的规划内容、优先级和时间线。

### 核心价值

- **透明化规划**：让团队成员了解项目发展方向
- **优先级管理**：明确功能开发顺序和资源分配
- **预期管理**：让用户了解未来功能的时间线
- **协作基础**：为需求讨论和计划调整提供基础

### 适用场景

✅ 需要创建 ROADMAP 的项目：
- 有明确发展计划的持续维护项目
- 多人协作的项目
- 需要向用户展示未来规划的开源项目
- 有季度/年度规划的企业项目

❌ 可以不创建 ROADMAP 的项目：
- 一次性使用的工具脚本
- 个人学习项目
- 已停止维护的项目

## 版本阶段划分

### 1. 进行中 (In Progress)

**定义**：当前正在开发或即将发布的版本

**内容要求**：
- 版本号（如 v1.0.0）
- 预计发布日期或时间范围
- 核心功能列表
- 阻塞问题（如有）

**示例**：
```markdown
## v1.0.0 (预计 2024 Q2)
- [x] 核心架构设计
- [ ] 用户认证模块 (进行中)
- [ ] 数据持久化 (进行中)
- [ ] API 接口开发
- 阻塞问题: 需要确定数据库选型
```

---

### 2. 计划中 (Planned)

**定义**：计划开发但尚未开始的功能

**内容要求**：
- 按版本分组
- 标注优先级（High/Medium/Low）
- 可选时间范围

**示例**：
```markdown
## v1.1.0 (计划中 - 2024 Q3)
### 高优先级
- 支持多语言国际化
- 用户角色权限管理

### 中优先级
- 数据导出功能
- 邮件通知系统

### 低优先级
- 主题定制功能
- 插件系统
```

---

### 3. 未来规划 (Future)

**定义**：长期愿景和探索性功能

**内容要求**：
- 按领域分组（不绑定版本）
- 描述愿景而非具体实现
- 可选技术方向

**示例**：
```markdown
## 未来规划

### AI 能力增强
- 集成 LLM 实现智能推荐
- 自然语言查询接口
- 自动化测试生成

### 性能优化
- 分布式缓存架构
- 数据库分片策略
- 异步任务队列
```

---

### 4. 已废弃 (Deprecated)

**定义**：已取消开发的功能或计划

**内容要求**：
- 功能名称
- 废弃原因
- 替代方案（如有）

**示例**：
```markdown
## 已废弃

### 移动端 App
- **废弃原因**: 资源有限，优先级调整
- **替代方案**: 响应式 Web 界面
- **废弃日期**: 2024-01-15
```

## 条目格式规范

### 功能条目格式

```markdown
- [功能描述] (状态)
  - 详细说明
  - 负责人: @username
  - 优先级: High/Medium/Low
```

**状态标记**：
- `[ ]` 待开始
- `[~]` 进行中
- `[x]` 已完成

### 示例

```markdown
- [用户认证系统] [~]
  - 支持邮箱和手机号登录
  - JWT Token 认证
  - 负责人: @zhangsan
  - 优先级: High
```

---

### 版本条目格式

```markdown
## [版本号] (时间范围/预计日期)

### 高优先级
- [功能 1] (状态)
- [功能 2] (状态)

### 中优先级
- [功能 3] (状态)

### 低优先级
- [功能 4] (状态)

### 技术债务
- [重构项 1] (状态)
- [优化项 2] (状态)
```

## 维护策略

### 更新频率

- **进行中版本**：每周更新
- **计划中版本**：每月更新
- **未来规划**：每季度更新

### 更新触发条件

1. **版本发布时**：
   - 将"进行中"版本移到"已完成"
   - 将"计划中"版本移到"进行中"
   - 添加新的"计划中"版本

2. **需求变更时**：
   - 更新功能描述或优先级
   - 添加或删除功能项

3. **季度规划时**：
   - 审查"未来规划"
   - 调整计划中版本的内容

4. **功能废弃时**：
   - 将功能移到"已废弃"
   - 说明废弃原因

### 与 CHANGELOG 的关系

| 维度 | ROADMAP | CHANGELOG |
|------|---------|-----------|
| 时间 | 面向未来 | 面向过去 |
| 内容 | 计划和规划 | 已完成的变更 |
| 版本 | 未来版本 | 已发布版本 |
| 状态 | 可变 | 不可变 |

**同步规则**：
- 版本发布后，将 ROADMAP 中该版本的内容复制到 CHANGELOG
- 更新功能状态（`[ ]` → `[x]`）
- 保持变更类型一致（Added/Changed/Removed/Fixed）

## 最佳实践

### ✅ 推荐做法

1. **保持现实性**：
   - 只记录有明确计划的功能
   - 避免过度承诺
   - 定期清理不再适用的条目

2. **明确优先级**：
   - 使用 High/Medium/Low 标记
   - 每个版本不超过 3-5 个高优先级项
   - 优先级随时间动态调整

3. **标注负责人**：
   - 每个功能至少一个负责人
   - 避免责任模糊

4. **保持更新**：
   - 定期检查和更新状态
   - 版本发布后及时同步

### ❌ 避免做法

1. **过度详细**：
   - 避免列出所有技术细节
   - 高层规划即可

2. **长期不变**：
   - 避免ROADMAP成为死文档
   - 定期审查和调整

3. **缺少状态**：
   - 避免不标注功能状态
   - 保持进度透明

4. **忽视废弃功能**：
   - 及时记录已废弃功能
   - 说明原因和替代方案
